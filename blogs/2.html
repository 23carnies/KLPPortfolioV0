BEM and Sass Basics

I love CSS. I've been working with it for a long time and it works for me. It can get unweildy, however, and I was excited to learn about organizational methodologies to keep it under control.

I have seen BEM out in the wild but never knew the name or concept behind it. I started learning Sass recently and realized that BEM and Sass work hand-in-hand. So I've been trying to get a better handle on both.

BEM

There are several organization methodologies for CSS, OOCSS, SMACSS, SUITCSS, Atomic, and BEM. The Block, Element, Modifier methodology of organization allows us to see the hierarchical relationships between CSS elements.

Block level selectors stand alone on the page. They are not dependent on other blocks or elements. Name these with class selectors using a meaningful and simple name.

                    
            .menu
            .form
            .picture-frame

Element level selectors are part of a block. They do not stand alone and are semantically linked to their block level. Name these with class selectors beginning with the block element + two underscores + the element descriptor.

                    
            .menu__item
            .form__search
            .picture-frame__img

Modifiers are extra classes that modify the block or element but keep their original class. Name these with class selectors beginning with the block or element name + two dashes + modifer descriptor

                    
            .menu__item--active
            .form--large
            .picture-frame__img--highlighted

The hierarchical structure of BEM limits cascading and specificity issues. The structure is easy for other programmers to understand. Also, using independent blocks can reduce the amount of code to maintain.

Sass and SCSS

Sassy CSS makes me so very happy. BEM lends itself easily to the Sass conventions. For me, Sass is what CSS has been striving to be all this time.

CSS preprocessors generate CSS from their own syntax and allow extra features not natively available in CSS. Sass, which stands for Semantically awesome style sheets, and its cousin SCSS (Sassy CSS) are similar with the main difference being the use of brackets versus indentation only in the syntax. I like the brackets and find SCSS a little easier to understand visually. Sass/SCSS can be installed by npm or yarn or you can use a VSCode extension like Live Sass Compiler.
The Live Sass Compiler has been my go to so far. Create a scss folder and main.scss inside of it. HTML still needs a CSS file to render styles. Click Watch Sass and the compiler creates the main.css file to link to the HTML. Only write code in the scss file. Each time the compiler runs, the .css file is overwritten.

Most CSS code works in SCSS. This is what makes SCSS easy to pick up. To change the background color of the body of the page: 

                        
                body {
                    background-color: #062F4F;
                }
Just the same as it is in CSS. The bonus points for Sass are the extras: Modularity, Nesting, Variables, Maps, Functions, and Mixins.

Variables

Sass allowed for variables before CSS did so natively. The syntax is slightly different. 

                        
                /* Set variables here */
                $primary-color: #062F4F;
                $accent-color: #AD6EA2;
                $text-color: #FFFFFF;

                /* Use them here */
                body {
	                background-color: $primary-color;
                }
Variables make it easy to change the value in one place, keeping the code simple and readable.

Nesting

This is where BEM comes in handy. Elements can be nested inside of their block components keeping selectors grouped together. There is a lovely shorthand for nested BEM named elements. The & ampersand indicates the same name as the parent component. 

                        
                .menu-nav {
                    @include centerFlex;
                    max-width: 1500px;
                                                
                    &__item {
                        list-style: none;
                        position: relative;
                        padding: .8rem;
                        transition: 0.5s;
                    }
                
                    &__item > a {
                        text-decoration: none;
                        padding: .5rem 1rem;
                        color: $text-color;
                        font-size: 20px;
                        font-size: 1.5rem;
                        border-radius: .5rem;
                    }
                    
                    #name {
                        font-family: 'Eczar', serif;
                        font-size: 2rem;
                    }

Here you can see the block and element selectors were named ".menu-nav" and ".menu-nav__item". There is then an < a > tag inside the element ".menu-nav__item" and one has an id of name. These are all nested under their parent element. Easy to find; easy to read.

Extends

Sometimes a class needs all of the styling of another class plus some. Instead of writing the same code again or mucking up the HTML with extra elements, @extend can keep your CSS and HTML clutter free.

                        
            .btn {
                padding: 10px;
                color: #000;
                border-radius: 8px;
                background-color: green;

                &--danger {
                    @extend .btn;
                    background-color: red;
                }
            }

The specificity of the extended class will overwrite the differences from the class extender. Extends will only work on the class it is added to. It will not pass down to the inner selector ( &__inner ).

Functions and Mixins

Functions and Mixins appear similar in design but are slightly different. Mixins output Sass code that will compile to CSS while Functions return a value to be used for a CSS property or be passed into another function or mixin. Functions are used to compute and return values whereas Mixins are used to define styles.

Mixins

Mixins make reuseable code shortcuts that can be used throughout your project. Use flexbox to center often? You can make an easy Mixin for that! 

                        
                    /* Create Mixin here */
                    @mixin flex-center {
                        display: flex;
                        justify-content: center;
                        align-items: center;
                    }
                    
                    /* Call your Mixin here */
                    .box {
                        @include flex-center;
                    }

                    /* Mixins can also take a variable */
                    @mixin flex-center-variable($flex-direction) {
                        display: flex;
                        justify-content: center;
                        align-items: center;
                    }

                    /* Called here with the variable */
                    .box {
                        @include flex-center-variable(column)
                    }

Functions

This function example is taken from the Sass-lang site.

                    
                /* Create function */
                @function invert($color, $amount: 100%) {
                    $inverse: change-color($color, $hue: hue($color) + 180);
                    @return mix($inverse, $color, $amount);
                  }
                  
                /* Call function */
                .header {
                  background-color: invert($primary-color, 80%);
                }

In naming Mixins and Functions in Sass, hyphens and underscores are equivalent. So flex_center and flex-center would refer to the same funciton or mixin.

Modularity

For very large projects, SCSS can be divided into multiple files. Create a file for variables, one for mixins, or one for each different page. These files are kept in the same scss directory, named beginning with an underscore, and connected to the main SCSS file using @import.
So a file _config.scss may contain all the variables and mixins. Connect to main.scss: 

                        
            @import 'config'
